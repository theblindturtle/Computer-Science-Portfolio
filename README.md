# CS 230 Software Design Document
## Project Summary
The client for this project was The Gaming Room, a company that wanted to expand its game, Draw It or Lose It, from an Android-only application into a web-based application that could support multiple operating systems and devices. The software needed to support multiple teams and players while ensuring that game names and team names remained unique. The application also needed to support a distributed environment and provide a consistent experience across platforms.

## What I Did Well
One area I think I did well was evaluating different operating systems and recommending the best platform for the client. I compared Linux, Windows, macOS, and mobile platforms while considering factors such as cost, performance, scalability, and security. I also organized the software design document in a way that clearly addressed the client's requirements.

## Helpful Parts of the Design Process
Working through the design document helped me better understand the importance of planning before development begins. Creating the document required me to think about requirements, operating platforms, storage management, memory management, networking, and security before any code was written. This process helped create a clearer roadmap for how the software should be designed.

## What I Would Revise
If I could revise one part of the project, I would spend more time developing the system architecture section. Adding more diagrams and technical details would help better illustrate how the various components communicate with one another in a distributed environment.

## Understanding User Needs
I interpreted the user's needs by focusing on The Gaming Room's goal of expanding Draw It or Lose It to multiple platforms while maintaining a consistent experience for all users. This included evaluating different operating systems, recommending Linux as the server platform, and designing a solution that would support desktop and mobile users. Understanding user needs is important because successful software must solve the client's problem while remaining easy for users to access and use.

## My Approach to Software Design
My approach to software design was to first understand the requirements and then evaluate possible solutions based on those requirements. Throughout this course I learned the importance of considering scalability, maintainability, performance, and security early in the design process. In future projects I would continue using design documents, system modeling, and platform evaluations to help analyze requirements and create effective software solutions.
